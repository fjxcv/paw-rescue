## 第2页：项目背景与技术架构

### 2.1 流浪宠物救助痛点与平台定位
- 当前流浪动物信息分散、缺乏统一发布渠道，救助过程不透明导致志愿者无法追踪进展，领养流程繁琐缺乏在线申请机制。平台定位为连接"发现救助→医疗恢复→领养安置"全流程系统。本人负责宠物领养模块与救助追踪模块的前台功能开发与测试，不包含管理员后台审核功能。

### 2.2 前后端分离架构与关键技术设计
- **后端**：Python 3.11 + Django 6.0 + DRF + JWT + SQLite。ViewSet路由与自定义action、三层权限类体系、救助状态机由 `STATUS_FLOW` 字典严格约束流转
- **前端**：React (CRA) + React Router + Axios + Bootstrap 5。ProtectedRoute/AdminRoute路由守卫、Axios拦截器自动注入JWT、组件loading/error/empty三态处理、高德地图API（逆地理编码+地址自动补全）

## 第3页：宠物领养模块 —— 宠物发现与多维度筛选

### 3.1 列表浏览与多条件组合筛选
- 卡片网格展示待领养宠物：照片、名称、品种、月龄、性别、地区、体型、健康状态
- 多条件AND组合筛选：物种(6种下拉)、性别、年龄(幼龄/青年/成年/高龄+自定义月龄)、国家→省→市三级联动地区（选项基于已有数据动态生成）、关键词按名称回车搜索（非实时）
- 后端过滤：非管理员仅展示 `is_public=True` 的宠物，一键重置清空所有条件

### 3.2 附近搜索与宠物详情
- **附近搜索**：浏览器Geolocation API获取GPS → 高德逆地理编码得省/市 → 后端Haversine公式计算球面距离 → 返回同省指定半径内宠物 → 卡片展示距离(如"距您3.2km")、按距离排序 → 支持半径切换(3/5/10/20/50km)
- **宠物详情**：大图多图前后翻页+位置指示器、完整信息展示(种类/性别/年龄/体型/地区/健康)、Markdown格式描述渲染、不可领养宠物显示提示并隐藏申请按钮

## 第4页：宠物领养模块 —— 领养申请与状态追踪

### 4.1 领养申请三步流程与冲突检测
- **入口控制**：ProtectedRoute路由守卫，已登录→/adopt/{petId}，未登录→/login；宠物非available状态隐藏"带我回家"按钮
- **Step1-问卷**：10字段(姓名/地址/电话/微信/性别/年龄/住宅类型/住房情况/户外区域多选/同意/经验/变化)，条件显隐逻辑("其他"→输入框、"是"→描述框)，客户端校验(必填、电话格式、年龄0-150)。API：`POST /adopt/applications/` → `POST questionnaire/`
- **Step2-附件**：身份证+居住证明，两段式上传(先`uploadAPI.upload()`到/uploads/获取URL，再`addAttachment()`关联)，前端校验(jpg/png/pdf、≤5MB)
- **Step3-成功**：动画+"1-3个工作日审核"提示，宠物状态 available→pending
- **冲突检测**：同用户同宠物不可有进行中(pending/approved)的重复申请；已有他人申请中的宠物互斥保护

### 4.2 申请状态追踪与七级状态体系
- 我的申请列表(表格展示+分页10条/页)，仅展示本人申请(后端按applicant过滤)
- **7种状态映射**：pending黄-待审核 / rejected红-审核拒绝+详情 / need_material黄-待补充 / approved绿-审核通过 / approved+scheduled黄-待核验 / approved+passed绿-核验通过 / approved+failed红-核验失败+详情
- 拒绝/失败详情页（红色渐变头部+audit_opinion/verify_note+时间），getDetailType区分类型，越权保护403

## 第5页：救助追踪模块 —— 信息上报与状态机

### 5.1 信息上报与智能定位
- **表单**：昵称(≤50字)、联系方式(≥5字符)、发现地址、体型(小/中/大)、健康(健康/轻微伤病/严重伤病)、怕人(是/否)、照片(选填，多张jpg/png/pdf≤5MB)。appearance字段前端自动合并
- **GPS定位流程**：`getCurrentPosition()` → 高德逆编码→"省+市+区+街道"自动填充 → CoordinateField保留6位小数。异常处理：拒绝权限→提示手动输入、超时15s、无法获取→降级方案
- **编号生成**：`RES{YYYYMMDD}{3位流水号}`，查询当天最大编号+1，每日重置。提交后自动创建首条status_log(from=None→pending_rescue)

### 5.2 救助状态机与流转约束
- **6种状态**：pending_rescue(待救助/初始)、in_medical(医疗中)、recovering(恢复中)、awaiting_adoption(待领养)、rescued(救助成功/终态)、abandoned(已终止/预留)
- **流转规则(STATUS_FLOW)**：`pending_rescue --help--> in_medical --update_status--> recovering --update_status--> awaiting_adoption --update_status--> rescued`
- **关键约束**：pending_rescue仅能通过help接口自动推进；状态只能单向邻步流转(禁止跳跃/回退)；rescued为终态不可变更；每次手动变更必须填写备注(前后端双重校验)

## 第6页：救助追踪模块 —— 协作救助与时间线

### 6.1 多人协作救助与阶段记录
- **响应救助(help)**：`POST /rescue/cases/{id}/help/`，用户→helpers多对多，首次响应记录help_date+自动推进pending_rescue→in_medical+弹窗显示上报人联系方式，后续响应仅追加helpers不重复推进。防重复：后端`user in helpers`检查+前端localStorage双重防护
- **状态更新(update_status)**：仅helper或admin可操作，前端显示当前状态(只读)+切换时自动显示下一合法状态，必须填写备注→`PATCH status/`，每次变更创建RescueStatusLog(from→to+operator+remark)
- **阶段记录(stage_records)**：救助人随时填写，支持多次、按时间倒序、含操作人+时间+内容，前后端双重content非空校验

### 6.2 时间线整合与可视化追踪
- **数据整合**：status_logs(状态变更日志)+stage_records(阶段记录)，页面并行请求两数据源，前端按时间戳合并
- **可视化呈现**：页面顶部横向流程图(已过节点高亮/当前加粗/未达灰显/终态特殊标识)，点击节点切换时间线；黄色图标=状态变更("状态变更为『医疗中』：xxx响应救助")，蓝色图标=阶段记录("今日体检正常")
- **辅助功能**：悬停节点Tooltip显示预计停留时长(待救助1-3天/医疗中7-30天/恢复中7-14天/待领养7-60天)；按编号精确搜索(大小写不敏感)，历史存localStorage按用户隔离最多3条

## 第7页：权限控制体系

### 7.1 三层权限架构与数据隔离
- **L1-公开(AllowAny)**：宠物列表/详情、救助列表/详情
- **L2-登录+活跃(IsAuthenticated+IsActiveUser)**：提交领养申请、上报救助、响应救助
- **L3-授权(IsAdminRole/is_helper)**：宠物管理、救助状态变更、阶段记录
- **前端双重防护**：ProtectedRoute检查token→跳转/login；AdminRoute检查role→跳转/pets；操作按钮按角色条件渲染
- **数据隔离**：领养申请按applicant过滤，救助按helpers/reporter过滤，越权返回403
- **封禁机制**：profile.status==1时IsActiveUser拒绝所有写操作，读操作不受影响

## 第9页：开发测试中的典型问题

### 9.1 关键技术问题与解决方案
- **文档生成编码问题**：中文双引号`""`(U+201C/U+201D)在文件写入时→ASCII`"`(U+0022)→Python语法错误。解决：使用角落括号`【】`替代所有中文双引号
- **旧版.doc格式兼容**：`python-docx`不支持OLE2二进制格式。解决：`olefile`解析结构+UTF-16LE解码提取文本
- **前端路由与后端权限不一致**：救助列表/详情后端AllowAny但前端包裹ProtectedRoute→未登录用户无法通过UI访问但可通过API直接访问。建议：统一前后端权限策略
- **abandoned状态不可达**：模型定义了abandoned终态但无任何API路径可达，属业务死状态，需后续迭代完善流转逻辑
